ABSTRACT

The evolution of web security from HTTP to HTTPS represents a critical
transformation in protecting digital communications. This research
examines the fundamental security mechanisms underlying HTTPS, with
particular emphasis on cryptographic protocols, certificate management,
and HSTS enforcement. Through practical implementation and vulnerability
analysis, this study demonstrates the effectiveness of HSTS in
preventing SSL stripping attacks a prevalent man-in-the-middle attack
vector that downgrades secure connections to exploit plaintext HTTP
communication.

The research employs a mixed-methods approach combining literature
review with hands-on security testing using Apache web server,
mitmproxy, and Kali Linux environments. Our experimental setup simulates
real-world attack scenarios to evaluate the defensive capabilities of
HSTS against protocol downgrade attacks. The findings reveal that while
HTTPS provides essential encryption through TLS/SSL protocols, proper
implementation of HSTS policies significantly enhances security posture
by eliminating the initial HTTP connection vulnerability.

Key contributions include: a comprehensive analysis of HTTPS
cryptographic foundations, practical demonstration of SSL stripping
attack vectors, empirical evaluation of HSTS effectiveness, and
comparative analysis of web security implementations. The research
concludes that HSTS preloading represents a critical defence mechanism
for modern web applications, though implementation challenges remain in
legacy systems and emerging quantum computing threats.

II) INTRODUCTION

The modern internet has become an indispensable infrastructure for
global communication, commerce, and information exchange. However, this
digital transformation has also introduced significant security
challenges that threaten the confidentiality, integrity, and
authenticity of online communications. The Hypertext Transfer Protocol
(HTTP), originally designed for simple document retrieval in the early
1990s, was never intended to handle the sensitive data transactions that
characterize today's web applications.

The fundamental security limitation of HTTP its plaintext nature creates
vulnerabilities that malicious actors can exploit. This inherent
insecurity led to the development of HTTPS (HTTP Secure), which layers
the HTTP protocol over Transport Layer Security (TLS) or its
predecessor, Secure Sockets Layer (SSL), providing encryption,
authentication, and data integrity.

Despite the widespread adoption of HTTPS, security vulnerabilities
persist, particularly in the form of protocol downgrade attacks such as
SSL stripping. These attacks exploit the common user behaviour of
accessing websites without explicitly specifying the “https://"
protocol, allowing attackers to maintain insecure HTTP connections while
intercepting and potentially modifying sensitive data.

III. LITERATURE REVIEW

A) HTTP:

To understand the evolution of web security, one must first understand
the problem HTTP was created to solve. Before the World Wide Web, the
internet consisted of fragmented systems and isolated protocols, which
led to an inconsistent and inefficient user experience. There was no
single, unifying protocol designed for distributed, hypermedia
information systems.[5]

HTTP was born out of this necessity, developed in the early 1990s by Tim
Berners-Lee as part of the World Wide Web project at CERN [1]. The
earliest HTTP version, HTTP/0.9, was a simple protocol supporting only
the GET method and returning raw HTML without headers [2]. Building on
this, HTTP/1.0 [2] added support for headers, multiple methods (GET,
POST, HEAD), and status codes, enabling richer client-server
interactions. Later, HTTP/1.1 [2] introduced persistent connections,
host headers, and caching mechanisms, becoming the backbone of modern
web communication. Figure 1 presents a structured overview of HTTP’s
evolution, offering a comparative analysis of its major versions and
their respective advancements.

HTTP was designed as a stateless, plaintext, application-layer protocol
to serve as a universal gateway, allowing a single client (browser) to
access diverse resources. However, it was built for simplicity, not
security. All communication under HTTP is transmitted in plaintext,
meaning any intermediary can intercept or alter data in transit. HTTP
provides no inherent confidentiality, no server authentication, and no
data integrity verification [2].

As the web began handling sensitive data, such as financial transactions
and personal information, these vulnerabilities became critical. This
led to the development of HTTPS, which layers HTTP over the Transport
Layer Security (TLS) protocol.[3][4]

 C) The Evolution of SSL/TLS: A History of Hardening the Web

As HTTP evolved into the foundation of the modern web, it became clear
that a plaintext protocol could not support the growing need for secure
transactions. The solution came in the form of cryptographic protocols
designed to secure communication channels beginning with SSL.

SSL was first developed by Netscape Communications in the early 1990s,
shortly after Tim Berners-Lee released HTTP and the World Wide Web into
the public domain [1]. As online commerce and data exchange grew, it
became evident that HTTP’s plaintext structure posed a critical security
risk. SSL aimed to solve this by creating an encrypted tunnel around the
HTTP session.

The earliest version, SSL 1.0, was never released due to serious design
flaws [2]. It lacked strong authentication, had vulnerabilities in key
negotiation, and was susceptible to basic attacks. These shortcomings
led to a rapid redesign.

SSL 2.0, released in 1995, introduced public-key encryption and
handshake-based key exchange. However, it suffered from:

-   No protection for the handshake itself,

-   Weak cipher negotiation (no downgrade protection),

-   And vulnerabilities to man-in-the-middle (MITM) attacks [3].

In response, SSL 3.0 emerged in 1996, redesigned by Paul Kocher and
Netscape. It introduced:

-   Authenticated handshake messages using MACs,

-   Better cipher suite negotiation,

-   Support for stronger key exchange algorithms like DHE.

SSL 3.0 became the basis for the formal standardization of secure
communication by the IETF, leading to the creation of the TLS protocol.

The Transport Layer Security (TLS) protocol was introduced in 1999 with
TLS 1.0, based on SSL 3.0 but with more robust cryptographic algorithms
[4].

Subsequent versions included:

Table 1: TLS Versions Comparision:

  Version       Year      Key Updates
  ------------- --------- -----------------------------------------------
  TLS 1.0       1999      Formalization of SSL 3.0 under IETF, added HMAC

                          forward secrecy [5]

Modern protocols like HTTP/2 and HTTP/3 are built with TLS as a
requirement. Most browsers and operating systems now enforce:

-   Minimum TLS 1.2

-   Strong cipher suites only

-   Certificate transparency and HSTS

IV) Application context:

A) SSL Stripping 

  is a MITM attack that downgrades a user’s secure HTTPS connection to
  HTTP, enabling the attacker to intercept or alter traffic. First
  introduced by Moxie Marlinspike in 2009, it exploits users’ habit of
  typing URLs without the “https://” prefix [11].

In this attack, the victim’s browser sends an initial HTTP request,
which the attacker intercepts (using methods like ARP spoofing or rogue
Wi-Fi) and responds with a fake HTTP site. Meanwhile, the attacker keeps
a real HTTPS connection with the server but strips encryption between
themselves and the victim, acting as a transparent proxy.[12]. Figure 5
represent typical flow

Figure 5: SSL Stripping Flow

[]

This lets them:

-   Steal login credentials and cookies

-   Inject malicious content

-   Hijack or monitor sessions

Though the server uses HTTPS, the victim’s browser never establishes a
secure connection and remains unaware.[11]

B) HSTS:

To prevent downgrade attacks like SSL Stripping, modern websites use
HSTS, a policy that forces browsers to connect only via HTTPS for
specified domains. [10,13] Once a browser receives an HSTS header, it
will:

-   Automatically upgrade all future HTTP requests to HTTPS

-   Reject untrusted certificates or insecure connections

-   Optionally enforce the policy on all subdomains

By removing the initial insecure HTTP request that SSL Stripping
exploits, HSTS effectively blocks this attack.

The policy is set via a response header, for example:

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
[10]

Security is further enhanced by submitting domains to the HSTS preload
list a browser-maintained registry that hardcodes sites to always use
HTTPS from the very first visit, eliminating the "first-connection
vulnerability" completely. [13]

To be added to the list, a website must:

-   Use a valid HTTPS certificate

-   Redirect all HTTP traffic to HTTPS

-   Serve the HSTS header with required directives:

-   Support HTTPS on all subdomains

Redirecting users from HTTP to HTTPS is a common way to secure web
traffic by encrypting data exchanged between the browser and the server.
However, this method alone leaves a critical gap known as the "initial
plaintext request" vulnerability. When a user first visits a site over
HTTP, the request is unencrypted and can be intercepted before the
redirect occurs.

This opens the door to SSL Stripping, where an attacker intercepts the
HTTP request and silently acts as a proxy maintaining a secure
connection with the server, but an insecure one with the user. As a
result, compromised.

To close this gap, sites must implement HSTS. HSTS tells browsers to
always use HTTPS for future visits, eliminating the chance for an
attacker to exploit the initial unencrypted request. While HTTPS
redirects are helpful, HSTS is essential for truly secure web
communication.

VIII) Conclusion:

This study exposed SSL‑stripping’s real‑world risk and empirically
validated HTTP Strict Transport Security (HSTS) as the definitive fix.
After outlining HTTP’s plaintext vulnerability and HTTPS/TLS’s encrypted
safeguards, we built a lab using Apache, Mitmproxy, and arpspoof to
perform a live Man‑in‑the‑Middle downgrade attack.

By intercepting a browser’s initial HTTP request and capturing
credentials in cleartext, we showed the flaw lies in that unprotected
redirect not in TLS cryptography itself. Enforcing HSTS closes this
“HTTP window,” forcing browsers to upgrade every connection to HTTPS
before sending data. This protection is especially vital on public
networks (e.g., cafes or airports), where ARP spoofing makes SSL
stripping trivial.

Although HSTS is the cornerstone defence, we found layered measures AP
isolation, host firewalls, and antivirus to thwart network‑level
threats. Operating under a self‑signed certificate in a controlled
environment, our work paves the way for future research into
post‑quantum TLS, advanced attack vectors, and deployment with trusted
CAs. For any organization serious about web security, a robust HSTS
policy is essential.

Section 2: Understanding Fundamentals/Key Terms:

Cryptography:

Cryptography derived from the Greek words kryptós (“hidden”) and
graphein (“to write”) is the scientific discipline focused on securing
communication via mathematical techniques. It translates plaintext into
ciphertext, ensuring that only authorized parties with the correct keys
can access the original message. Cryptography underpins critical aspects
of information security, including confidentiality, integrity,
authentication, and non-repudiation. [17,16]

Block ciphers work by dividing the data into fixed-size blocks (for
example, 128 bits) and encrypting each block separately using the same
key. If the final block is not the right size, padding is added. A
common example is AES (Advanced Encryption Standard).

Stream ciphers, on the other hand, encrypt data one bit or byte at a
time. They use something called a keystream, which is a series of
random-looking bits, and combine it with the original data using the XOR
operation. This method is fast and useful for real-time communication,
like video or audio calls.

Understanding some more terms:

Key Generation:

Key generation is the process of creating cryptographic keys used to
secure data through encryption and decryption. These keys ensure that
only authorized users can access the protected information.

The method used to generate keys is called a key generation algorithm.
Two main types of generation are: Symmetric and Asymmetric key
generation.

Key Size:

Key size, measured in bits, defines the length of a cryptographic key
and plays a crucial role in the overall strength of encryption.
Generally, larger keys offer stronger security by increasing the number
of possible combinations, making brute-force attacks more difficult.
However, greater key lengths require more computational resources.

Note: Bigger does not always mean better. The security also depends on
the algorithm itself. For instance, AES-128 is considered more secure
than RSA-1024 due to differences in their cryptographic design.

Section:5: Understanding Integrity terms:

Hash:

A cryptographic hash function maps data of any length to a fixed-size
output called a hash or digest. This process is one-way, making it
infeasible to reverse, and is designed to be collision-resistant,
meaning it's highly unlikely two inputs produce the same output.

In HTTPS, hash functions like SHA-256 ensure data integrity even slight
changes to a message yield a completely different hash. They also secure
digital certificates, verifying that the certificate content hasn't been
tampered with.

HMAC (Hash-Based Message Authentication Code)

HMAC combines a cryptographic hash function with a secret key to ensure
both integrity and authenticity. Only parties with the shared key can
create or verify the hash.

In TLS, which powers HTTPS, HMAC protects data in transit by detecting
tampering and confirming the source, securing communication from
eavesdropping or manipulation.

Digital Signatures

Digital signatures use hash functions and asymmetric encryption to
provide authenticity, integrity, and non-repudiation. In HTTPS, they're
embedded in X.509 certificates.

A certificate’s contents are hashed and encrypted with the issuer’s
private key. Clients verify it using the corresponding public key
ensuring the certificate is both unaltered and trustworthy.

Algorithms like RSA-PSS and Ed25519 are commonly used, forming the basis
for secure, authenticated connections and preventing MitM attack.

Section 6: Understanding Certificates Terms:

X.509 Certificates:

Before obtaining a certificate either self-signed or from a CA the
administrator must generate a key pair using a key generation algorithm.

The X.509 standard, developed by the ITUT, forms the cornerstone of
digital certification within PKI. An X.509 certificate securely binds a
public key to a verified identity such as that of a user, server, or
organization enabling reliable authentication and encrypted
communication.

Crucially, X.509 defines more than standalone certificates issued by
CAs; it establishes a layered trust model composed of:

1.  Root Certificates: Issued by trusted root CAs and pre-installed in
    operating systems and web browsers.

2.  Intermediate Certificates: Issued by subordinate CAs to extend trust
    from the root authority to end entities.

3.  End-Entity Certificates: Issued to servers or users, these are
    presented during secure exchanges to verify identity.

Together, these certificates form a trust chain, in which credibility
flows from the root CA through intermediaries to the end entity. We can
sign other entities too with self-signed.

Note: Beyond its hierarchical trust model, the X.509 standard
incorporates critical mechanisms that enhance certificate reliability
and enforce security policies within a PKI.

Certificate revocation:

Used for invalidating certificates before their expiration when they are
compromised due to private key leaks, CA breaches, or personnel changes.
Without revocation, attackers could misuse certificates to impersonate
trusted entities.

To address this, systems verify revocation status using CRLs or the
OCSP, ensuring only valid certificates are trusted.

Certificate extensions:

Which embed metadata specifying permitted uses and constraints such as
path length or policy restrictions. These extensions serve as built-in
safeguards, restricting certificates to their intended functions.

designed for specific purposes and validation scopes. Fundamentally, a
certificate serves to verify the authenticity and integrity of an

Certificate Authorities (CAs):

Is a trusted entity responsible for issuing digital certificates that
authenticate the identity of something. These certificates
cryptographically bind a public key to a verified subject.

The standard issuance process involves several steps:

-   The applicant generates a CSR and submits it to the CA.

-   The CA performs identity validation through standardized procedures.

-   Upon successful verification, the CA issues a digitally signed TLS
    certificate.

-   When clients connect to the website, their browsers verify the CA’s
    digital signature on the certificate.

-   If the signature is valid and trusted, a secure HTTPS session is
    established.

The Root CA’s certificate is self-signed and pre-installed within
operating systems and browsers. This Root CA servers as the ultimate
trust anchor, enabling trust chains.

Chain of trust:

The Chain of Trust is a fundamental PKI principle that secures
communication over untrusted networks like the Internet. It supports
SSL/TLS protocols by ensuring every digital certificate can be traced
back to a trusted CA validating the holder’s identity and enabling
encrypted, authenticated connections.

Section 8: Understanding Vulnerabilities:

MIMT attack:

Occurs when an attacker secretly intercepts and potentially alters
communication between two parties, such as a user and a website. By
positioning themselves between the two, the attacker can eavesdrop,
steal sensitive data, manipulate messages etc.

SSL Stripping:

  A type of MITM attack where a secure HTTPS connection is downgraded to
  an insecure HTTP one. [see upper MIMT meant]

POODLE

POODLE targets outdated SSL 3.0, exploiting weak padding in CBC
encryption. By manipulating encrypted blocks and analyzing server
responses, attackers decrypt session data byte by byte a slow but
effective method.

[]

Quantum Computing:

Is a new type of computing that uses the strange and powerful principles
of quantum physics to process information in ways classical computers
cannot. Unlike traditional computers that use bits (0s and 1s)

Quantum computers use quantum bits or qubits, Properties like:

-   superposition (where qubits can be 0 and 1 at the same time).

-   entanglement (where qubits become linked so the state of one
    instantly influences another).

::Quantum computers can explore many possibilities simultaneously.

This puts current internet security at risk because:

-   Shor’s Algorithm: Can quickly break the two main ways websites
    exchange secret keys RSA and ECC used in secure connections.

-   Grover’s Algorithm: Can speed up guessing the secret keys that
    scramble data, making encryption like AES-128 much weaker. That’s
    why experts suggest using AES-256, which is much harder to break.

If strong quantum computers arrive, hackers might be able to:

-   Decrypt old secure messages: By breaking the keys that protected
    them, enabling “HNDL” attacks that compromise past confidential
    communications.

-   Fake website: By breaking the digital signatures and forge it.

-   Make current encryption weaker.
